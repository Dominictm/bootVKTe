# Бот-секретарь для бронирования столов (Telegram + VK)

Дата: 2026-07-16

## Цель

Клиенты клуба пишут боту в Telegram или VK, бот сверяется с Google Calendar клуба
и показывает свободные слоты по одному из 3 столов, позволяет забронировать
слот (создаёт событие в календаре) и отменить свою бронь.

## Предметная область

- В клубе 3 стола. Все брони по всем столам лежат в **одном** Google Calendar.
- Стол определяется по префиксу в заголовке события: `"Стол {N}: {Имя}"`.
- Часы работы клуба: **10:00–22:00** (fixed, без вариаций по дням недели в v1).
- Длительность одного слота бронирования: **фиксированные 4 часа**. Клиент
  выбирает только время начала — окончание вычисляется автоматически
  (`start + 4ч`), последний доступный старт — тот, что укладывается в 22:00.
- Часовой пояс клуба конфигурируется (например `Europe/Moscow`).

## Архитектура

Общее ядро + два тонких адаптера (Node.js/TypeScript, один проект):

```
src/
  core/
    availability.ts   — расчёт свободных 4-часовых слотов в диапазоне 10:00–22:00
                         для стола на дату, на основе списка занятых интервалов
    calendar.ts        — обёртка над Google Calendar API (list events, insert, delete)
                         через Service Account
    booking.ts         — сценарий бронирования/отмены (платформонезависимая логика)
    tables.ts          — константы: 3 стола и их префиксы заголовков
  adapters/
    telegram/          — grammY-бот: клавиатуры, обработчики, маппинг core → Telegram UI
    vk/                 — vk-io бот: клавиатуры, обработчики, маппинг core → VK UI
  config.ts             — переменные окружения
  index.ts              — запускает оба адаптера параллельно (единый процесс)
```

Оба адаптера используют один и тот же `core` — логика вычисления свободных
слотов и создания/отмены брони пишется один раз и не дублируется между
платформами.

## Доступ к Google Calendar

- **Service Account**: создаётся в Google Cloud, календарь клуба расшаривается
  на email сервисного аккаунта (роль "Изменение мероприятий"). Не требует
  участия человека при работе бота, токен не протухает.
- Настройки: `GOOGLE_CALENDAR_ID`, путь к JSON-файлу credentials сервисного
  аккаунта, часовой пояс — через `.env`.

## Сценарий диалога (идентичен в Telegram и VK)

1. Клиент пишет боту (`/start` или любое сообщение) → главное меню кнопками:
   **«Забронировать стол»** / **«Мои брони»**.
2. **Забронировать** → бот показывает кнопки дат на ближайшие 7 дней
   (сегодня … +6).
3. Выбор даты → бот показывает 3 кнопки столов.
4. Выбор стола → бот запрашивает события календаря с этим префиксом на эту
   дату, вычисляет свободные времена начала 4-часовых слотов внутри 10:00–22:00
   и показывает их кнопками.
   - Если свободных слотов нет — сообщение об этом и предложение выбрать
     другую дату/стол.
5. Выбор времени → бот просит имя текстом → показывает подтверждение (стол,
   дата, время, имя) с кнопками «Подтвердить» / «Отмена».
6. **Подтверждение**:
   - Бот повторно проверяет, что слот всё ещё свободен (защита от гонки —
     кто-то мог забронировать параллельно). Если уже занято — сообщает об
     этом и возвращает к выбору времени.
   - Иначе создаёт событие в Google Calendar: заголовок `"Стол {N}: {Имя}"`,
     точное время начала/окончания; в `description` — скрытые метаданные
     `platform`, `userId`, `username` (нужны для последующей отмены).
   - Отвечает «Забронировано ✅» с деталями брони.
7. **Мои брони** → бот ищет в календаре будущие события, чей `description`
   содержит текущие `platform`+`userId` → показывает списком кнопок → выбор →
   подтверждение отмены → бот удаляет событие через Calendar API → «Бронь
   отменена».

Отдельная сущность "клиент" не хранится — вся необходимая информация (кто
забронировал) живёт в `description` события календаря. Отдельной БД в v1 нет.

## Админ-панель

- Реализуется как отдельная команда в том же боте (`/schedule`), доступна на
  обеих платформах (Telegram и VK).
- Админ определяется по Telegram/VK user ID, заданному в `.env`
  (`ADMIN_TELEGRAM_ID`, `ADMIN_VK_ID`). Если команду вызывает не админ —
  бот отвечает как на неизвестную команду / игнорирует.
- По команде бот выводит текстом расписание броней на ближайшие 7 дней по
  всем 3 столам, сгруппированное по дням, внутри дня — по столам с временем
  и именем клиента:

  ```
  Пн, 17 июля
    Стол 1: 10:00–14:00 — Иван
    Стол 2: 14:00–18:00 — Мария
  Вт, 18 июля
    Свободно
  ```

- Данные берутся тем же способом, что и для «Мои брони» — чтением событий
  календаря за нужный диапазон дат, без отдельного хранилища.

## Развёртывание

- Один Node-процесс (`index.ts`), стартующий оба адаптера параллельно
  (`bot.launch()` для Telegram через grammY, `vk.updates.start()` для VK),
  оба через long polling.
- Запуск и автоперезапуск — через **systemd-юнит** на VPS
  (`club-bot.service`, `Restart=always`). В репозитории — файл юнита и
  инструкция по установке (копирование в `/etc/systemd/system/`,
  `systemctl daemon-reload && systemctl enable --now club-bot`).
- `README.md` с инструкцией: создание Service Account в Google Cloud,
  шаринг календаря клуба на его email, заполнение `.env`
  (`TELEGRAM_BOT_TOKEN`, `VK_TOKEN`, `GOOGLE_CALENDAR_ID`,
  `GOOGLE_APPLICATION_CREDENTIALS`, `TIMEZONE`), сборка и установка юнита.

## Обработка ошибок

- Если Google Calendar API недоступен или превышена квота — бот отвечает
  клиенту «Сервис временно недоступен, попробуйте позже» и логирует ошибку.
  Сложных ретраев/очередей в v1 нет.

## Тестирование

- Unit-тесты на `core/availability.ts` — расчёт свободных слотов, включая
  граничные случаи (слот впритык к границе 22:00, полностью занятый день,
  пересекающиеся события, событие ровно на границе слота).
- Остальное — ручная проверка через реальных ботов в Telegram/VK (внешние
  API не мокаются).

## Веб-панель настройки и запуск (добавлено 2026-07-16)

Помимо `/schedule` в самом боте, нужна отдельная веб-панель для настройки
подключения ботов (не для управления бронями).

- **Процесс**: тот же `index.ts` при старте поднимает Express-сервер
  веб-панели на порту **8546** (настраивается через `.env`, переменная
  `PORT`) и параллельно пытается запустить Telegram/VK-адаптеры на основе
  `config.json`. Если `config.json` отсутствует или в нём нет токена —
  соответствующий адаптер не стартует, в лог пишется предупреждение, но
  веб-панель работает в любом случае.
- **Хранение настроек**: `config.json` в корне проекта — `telegramBotToken`,
  `vkToken`, `googleCalendarId`, `googleApplicationCredentials`,
  `adminTelegramId`, `adminVkId`. Создаётся и перезаписывается веб-панелью
  целиком при сохранении формы. Добавлен в `.gitignore` (как и `.env`).
- **`.env`** сокращается до инфраструктурных секретов самой панели:
  `WEB_ADMIN_LOGIN`, `WEB_ADMIN_PASSWORD`, `PORT` (по умолчанию 8546).
- **Веб-панель**:
  - Один HTML-эндпоинт с формой, поля — все ключи `config.json`. Значения
    полей на странице предзаполняются текущим содержимым `config.json` (кроме
    отображения токенов не в открытом виде — не требуется в v1, простое
    текстовое поле допустимо, т.к. доступ уже защищён логином/паролем).
  - Basic Auth на всех маршрутах панели, логин/пароль сверяются с
    `WEB_ADMIN_LOGIN`/`WEB_ADMIN_PASSWORD` из `.env`.
  - Отдельная страница «Инструкция» с тем же текстом, что и в `README.md`
    (установка Service Account, получение токенов Telegram/VK, деплой на
    Windows и Linux).
  - После сохранения формы страница показывает подсказку: «Настройки
    сохранены. Перезапустите бота (stop → start), чтобы применить
    изменения» — автоматического релоада ботов в v1 нет.
- **Запуск/остановка процесса** — универсально для двух платформ хостинга:
  - Windows: `start.bat` (запускает `node dist/index.js` в фоне, PID
    сохраняется в `run.pid`) и `stop.bat` (читает `run.pid`, останавливает
    процесс через `taskkill`).
  - Linux: существующий `deploy/club-bot.service` (systemd) — без изменений.

## Вне рамок v1

- Разные часы работы по дням недели.
- Переменная длительность бронирования.
- Раздельные календари на стол.
- NLP/свободный текст для бронирования.
- Автоматический перезапуск ботов при изменении настроек в веб-панели.
- Шифрование/маскирование токенов в веб-панели (хранятся в открытом виде в
  `config.json`, доступ к панели защищён только логином/паролем).
